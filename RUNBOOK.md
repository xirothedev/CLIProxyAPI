# RUNBOOK

## Purpose and scope

This runbook is for operating CLIProxyAPI in production-like or self-hosted environments.

It focuses on:
- Startup modes and lifecycle
- Config, secrets, and management access
- Logging and troubleshooting
- Storage backend behavior
- Incident response and recovery checklist

It does **not** redefine product behavior. Source-of-truth remains the runtime code and config schema.

## Architecture snapshot

CLIProxyAPI is a single Go service that exposes:
- OpenAI-compatible routes under `/v1`
- Gemini-compatible routes under `/v1beta`
- Management control plane under `/v0/management` (only when management auth is configured)
- OAuth callback endpoints on the same server port

Core startup path:
- `internal/cmd/run.go` → `StartService`
- `internal/api/server.go` → route registration and management route gating

## Startup modes

### Local binary

Build and run:

```bash
go build -o cli-proxy-api ./cmd/server
./cli-proxy-api
```

Common runtime flags:
- `-config <path>`: config file path
- `-tui`: start terminal management UI
- `-standalone`: embedded local server for TUI mode

Service shutdown is graceful via SIGINT/SIGTERM.

### Docker / docker-compose

Docker image defaults:
- Binary path: `/CLIProxyAPI/CLIProxyAPI`
- Working directory: `/CLIProxyAPI`
- Exposed port: `8317`

Compose defaults (`docker-compose.yml`):
- Ports: `8317`, `8085`, `1455`, `54545`, `51121`, `11451`
- Config mount: `./config.yaml` → `/CLIProxyAPI/config.yaml`
- Auth mount: `./auths` → `/root/.cli-proxy-api`
- Log mount: `./logs` → `/CLIProxyAPI/logs`

### Embedded SDK service

The repository also supports embedding the service through `sdk/cliproxy`.

Operationally, embedded mode uses the same runtime routing/auth/logging model, with host process lifecycle controlling context cancellation.

### Cloud standby behavior (no config yet)

When `DEPLOY=cloud` and valid config is absent, service enters standby and does **not** start API routes:
- waits for shutdown signals
- logs cloud-standby message

## Configuration and secrets

Primary configuration file:
- `config.yaml` (template from `config.example.yaml`)

Key fields:
- `host`, `port`, `tls.*`
- `auth-dir`
- `api-keys`
- `remote-management.*`
- `logging-to-file`, `logs-max-total-size-mb`, `error-logs-max-files`
- retry/routing controls (`request-retry`, `max-retry-interval`, `routing.strategy`)

### Management access controls

Management routes are registered only when at least one is present:
- `remote-management.secret-key` in config
- `MANAGEMENT_PASSWORD` in environment
- local password in runtime mode (e.g. embedded/TUI standalone)

Management auth rules:
- All management requests require a key (`Authorization: Bearer <key>` or `X-Management-Key`)
- Remote access additionally depends on `remote-management.allow-remote` unless env override is active
- Failed remote attempts are tracked per IP with temporary ban behavior

### Secret handling notes

- If `remote-management.secret-key` is plaintext, runtime hashes it and writes back hashed form.
- Keep config and auth directories mounted persistently in containerized deployments.

### Optional alternate storage backends

Server can bootstrap config/auth from:
- PostgreSQL store (`PGSTORE_*` env)
- Git-backed store (`GITSTORE_*` env)
- Object storage (`OBJECTSTORE_*` env)

All backends maintain a local workspace mirror used by existing file-based flows.

## Logging and troubleshooting

### Logging modes

- `logging-to-file: false` → logs to stdout
- `logging-to-file: true` → rotating file output (`main.log` + rotations)

Log directory resolution:
1. If `WritablePath()` is set, use `WritablePath()/logs`.
2. Otherwise start with `./logs`.
3. If `./logs` is missing or not writable, fallback to `<auth-dir>/logs` when auth-dir resolves successfully.

### Log retention / cleanup behavior

- `logs-max-total-size-mb > 0` enables periodic directory size enforcement
- cleaner removes oldest `*.log` / `*.log.gz` files to stay within total size limit
- active `main.log` is protected during size enforcement

`error-logs-max-files` controls count-based cleanup for error log files generated by request logging paths.

### Request logging behavior

Management endpoints are excluded from request-body logging paths to avoid leaking management secrets.

## Common operational tasks

### Inspect service logs (container)

```bash
docker compose logs -f cli-proxy-api
```

### Inspect file logs (host mount)

```bash
ls -lah ./logs
```

### Clear logs via management API

```bash
curl -sS -X DELETE \
  -H "Authorization: Bearer <MANAGEMENT_KEY>" \
  http://127.0.0.1:8317/v0/management/logs
```

### Read logs incrementally via management API

```bash
curl -sS \
  -H "Authorization: Bearer <MANAGEMENT_KEY>" \
  "http://127.0.0.1:8317/v0/management/logs?limit=200"
```

### Retrieve request log by request ID

```bash
curl -sS -OJ \
  -H "Authorization: Bearer <MANAGEMENT_KEY>" \
  http://127.0.0.1:8317/v0/management/request-log-by-id/<REQUEST_ID>
```

### Validate management access

```bash
curl -sS \
  -H "Authorization: Bearer <MANAGEMENT_KEY>" \
  http://127.0.0.1:8317/v0/management/config
```

## Incident / recovery checklist

1. **Confirm process state**
   - Binary/service running?
   - Correct port bound?

2. **Check config load path**
   - Correct `config.yaml` path/mount?
   - In cloud mode, ensure config is valid (not empty/invalid)

3. **Validate management gating**
   - Is `remote-management.secret-key` (or env/local equivalent) set?
   - If remote access is needed, verify `allow-remote` and network path

4. **Check auth and storage state**
   - `auth-dir` mounted and writable
   - If using Postgres/Git/Object store, verify bootstrap/connectivity

5. **Inspect logs and request traces**
   - stdout/container logs or file logs
   - management log endpoints for targeted diagnostics

6. **Recover safely**
   - Keep persistent volumes intact (config/auth/logs)
   - Reapply known-good config
   - Restart process and re-check `/` then management health
